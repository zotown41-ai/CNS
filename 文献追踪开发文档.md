# 文献追踪开发文档

## 1. 项目目标

将两个已有项目的最新产物统一发布到一个可直接用于 GitHub Pages 的静态仓库中：

- 周报来源：`D:\vibecoding\文献追踪项目\CNS周报`
- 日报来源：`D:\vibecoding\文献追踪项目\paper-overview-extractor-RSS`
- 发布仓库：`D:\vibecoding\文献追踪项目\文献追踪`
- 远程仓库：`https://github.com/zotown41-ai/CNS.git`

目标是形成一个稳定的发布入口：首页展示导航，子目录分别提供周报与日报页面，并支持后续一键更新与推送。

## 2. 初始整理过程

### 2.1 环境检查

先检查了本机 Git / GitHub CLI 可用性，并定位两个源项目中的最新报告目录。

结论：

- `git` 可用
- `gh` 未安装
- 因此 GitHub 仓库操作主要通过 `git remote` + `git push` 完成，而不是通过 GitHub CLI 创建或管理

### 2.2 发布仓库初始化

在 `D:\vibecoding\文献追踪项目\文献追踪` 下建立发布仓库结构，包含：

- 根目录 `index.html`
- 根目录 `README.md`
- `周报/`、`日报/` 备用目录
- `[周报]/`、`[日报]/` 正式目录
- `.nojekyll`

其中首页用于作为 GitHub Pages 入口，链接到周报和日报子目录。

### 2.3 最新报告接入

从两个来源项目中拷贝最新产物到发布仓库：

#### 周报来源

来自 `CNS周报/reports/` 下最新目录中的：

- `*.html`
- `*.md`
- `*.json`

拷贝到：

- `[周报]/index.html`
- `[周报]/weekly_report.md`
- `[周报]/weekly_records.json`

#### 日报来源

来自 `paper-overview-extractor-RSS/output/` 下最新目录中的：

- `weekly_report.html`
- `weekly_report.md`
- `weekly_records.json`

重命名后拷贝到：

- `[日报]/index.html`
- `[日报]/daily_report.md`
- `[日报]/daily_records.json`

这里保留了“日报目录中的原始文件名仍以 weekly_ 开头”的兼容逻辑，并在发布时统一改成 daily_ 命名，便于前端访问与目录语义保持一致。

## 4. 问题定位与修复

### 4.1 Git safe.directory 问题

由于仓库目录所有者与当前沙箱用户不同，直接运行 Git 会报：

- `fatal: detected dubious ownership in repository`

处理方式：在执行 Git 命令时统一附加：

```powershell
git -c safe.directory=D:/vibecoding/文献追踪项目/文献追踪 -C D:\vibecoding\文献追踪项目\文献追踪 <git subcommand>
```

这样无需改动全局配置，就可以在当前流程中安全操作仓库。

### 4.2 重复目录问题

确认正式发布目录应为：

- `[周报]/`
- `[日报]/`

因此把以下重复目录从 Git 跟踪中移除：

- `周报/`
- `日报/`

对应提交：

- `704dc5a Remove duplicate publish directories`

修复后仓库结构与首页说明保持一致。

### 4.3 推送认证问题

在受限环境中直接推送时，曾出现：

- `schannel: AcquireCredentialsHandle failed: SEC_E_NO_CREDENTIALS`

原因是当前环境无法直接使用宿主机的凭据缓存。

处理方式：使用允许访问宿主凭据的 Git 推送执行方式完成推送。

## 5. 一键发布功能开发

为避免后续每次手工复制、改名、提交、推送，新增了一键发布功能。

### 5.1 新增文件

在发布仓库中新增：

- `scripts/publish_literature_tracking.ps1`
- `一键发布.bat`

### 5.2 脚本职责

`publish_literature_tracking.ps1` 负责完成以下工作：

1. 定位工作区与三个项目路径
2. 在计划发布模式下先检查目标日报是否存在，周一额外检查目标周报是否存在
3. 缺失时再触发对应生成脚本：
   - `CNS周报/run_weekly_report.bat`
   - `paper-overview-extractor-RSS/scripts/run_daily_report.ps1`
4. 在 09:15 的二次发布窗口优先校验目标内容是否已经发布，以及是否仍有待推送提交
5. 自动寻找并同步需要发布的报告目录
6. 自动复制并重命名发布文件到：
   - `[周报]/`
   - `[日报]/`
7. 自动更新子目录 README 与仓库根 README
8. 自动执行：
   - `git add -A`
   - `git commit`
   - `git push origin <branch>`

### 5.3 支持参数

脚本支持以下参数：

- `-SkipWeeklyGeneration`：跳过重新生成周报，只同步已有最新产物
- `-SkipDailyGeneration`：跳过重新生成日报，只同步已有最新产物
- `-SkipPush`：只提交本地，不推送
- `-AllowEmptyCommit`：允许空提交
- `-DailyOffset`：指定日报日期偏移
- `-ScheduledPublish`：启用计划发布编排逻辑
- `-PublishPhase Primary|VerifyRetry`：显式指定发布阶段
- `-TimezoneId`：指定计划发布的日期计算时区

### 5.4 批处理入口

`一键发布.bat` 用于双击启动，内部调用：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\publish_literature_tracking.ps1
```

这样非命令行场景下也可直接触发整套流程。

## 6. 编码与兼容性处理

### 6.1 UTF-8 要求

项目要求所有代码文件使用 UTF-8 编码，避免中文乱码。

在实现过程中：

- Markdown / HTML / bat / ps1 内容均按 UTF-8 写入
- 脚本中显式设置：

```powershell
[Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
$OutputEncoding = [System.Text.UTF8Encoding]::new($false)
$env:PYTHONIOENCODING = 'utf-8'
```

### 6.2 Windows PowerShell 5 对 UTF-8 脚本的兼容问题

在本地验证时发现，`powershell.exe`（Windows PowerShell 5）对“无 BOM 的 UTF-8 ps1”可能按系统编码解释，导致中文字符串被误读，进而引发解析错误。

处理方式：

- 所有脚本统一保持 UTF-8 编码
- 在脚本内显式设置控制台与子进程为 UTF-8
- 对需要双击运行的入口，优先通过 `powershell -File` 调用，避免中文按系统编码误解析

这是本次开发中最关键的兼容性修复之一。
